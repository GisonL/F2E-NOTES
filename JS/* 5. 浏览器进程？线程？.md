# 浏览器进程和线程

### 进程

​	进程就是操作系统中正在执行的程序，打开一个程序后，计算机会给这个程序分配独立的内存空间。

​	浏览器是多进程的，意思是你可以同时打开多个页面，每个页面都有自己的进程。如果浏览器是单进程的，当其中一个页面或插件奔溃后，整个浏览器就奔溃了。因此，浏览器是多进程的，进程之间互不干扰。

​	在浏览器中，主要有以下几个进程：

1. Browser进程：浏览器的主进程（协调、中控），只有一个，作用有
   1. 负责浏览器界面的显示，与用户的交互。前进后退等。
   2. 负责各个页面的管理，创建和销毁其他进程。
   3. 将Renderer进程得到的内存中的Bitmap位图，绘制到用户界面上。
   4. 网络资源的管理，下载等。

2. 第三方插件进程：每种类型的插件对应一个进程，仅当使用该插件时才会创建。
3. GPU进程：最多一个，用于3D绘制等。
4. 浏览器Renderer渲染进程(浏览器内核)：内部是多线程的。默认每个tab页面都有一个Renderer进程，互不影响。主要作用为
   1. 页面渲染，脚本执行，事件处理等。

简单来讲：在浏览器中打开一个网页相当于起了一个新进程(Renderer渲染进程)，进程内有自己的多线程。

>  对于我们前端来说，最重要的是渲染进程，渲染进程中包含处理不同任务的线程。

### 线程(浏览器内核，Renderer进程中)

​	浏览器的渲染进程是多线程的。主要常驻线程有：

1. GUI渲染线程
   1. 负责渲染浏览器界面，解析HTML、CSS，构建DOM输和RenderObject树，布局和绘制等。
   2. 当界面需要重绘或由于某种操作引发回流时，该线程就会执行。
   3. 注意，GUI线程和JS引擎线程是互斥的，当JS引擎执行时，GUI线程会被挂起，GUI更新会被保存在一个队列中，等到JS引擎空闲时立即被执行。

2. JS引擎线程
   1. 也成为JS内核，负责处理Javascript脚本程序。(例如V8引擎)
   2. JS引擎负责解析Javascript脚本，运行代码。
   3. JS引擎一直等待着任务队列中任务的到来，然后加以处理，一个Renderer进程中无论什么时候都只有一个JS线程在运行JS程序。
   4. 同样注意，GUI渲染线程和JS引擎线程是互斥的，如果JS执行的时间过长，这样就会造成页面的渲染不连贯，导致页面渲染加载阻塞。
   5. JS引擎中排列的任务形成执行栈。(遵循后进先出数据结构)

3. 事件触发线程
   1. 归属于浏览器而不是JS引擎，用来控制事件循环，可以理解为JS引擎忙不过来，需要浏览器另开线程协助。
   2. 当JS引擎执行代码块如setTimeout时(也可以来自浏览器内核的其他线程，如鼠标点击、AJAX异步请求等)，会将对应任务（事件绑定、回调函数）添加到事件线程中。
   3. 当对应的事件符合条件被触发时，该线程会把事件添加到待处理队列的队尾，等待JS引擎处理。(遵循先进先出的队列数据结构)
   4. 注意，由于JS的单线程关系（这里要区分渲染进程多线程和JS单线程的区别，能够执行JS脚本代码的只有JS引擎线程），所以这些待处理队列中的事件都得排队等待JS引擎处理(JS引擎空闲时才会去执行)

4. 定时器触发线程
   1. 传说中的setTimeout和setInterval所在线程
   2. 浏览器定时计数器并不是 由Javascript引擎计数的(因为Javascript引擎是单线程的，如果处于阻塞线程状态就会影响计时的准确)
   3. 因此通过单独线程来计时并触发定时(计时完毕后，将对应的任务添加到事件队列中，等待JS引擎空闲后执行)
   4. 注意，W3C在HTML标准中规定，要求setTimeout中地域4ms的时间间隔算为4ms。

5. 异步http请求线程
   1. 在XMLHttpRequest在连接后是通过浏览器新开一个线程请求
   2. 在检测到状态变更时，如果设置有回调函数，异步线程就产生状态变更事件，将这个回调再放入事件队列中。再由JS引擎执行。

### Broswer进程和浏览器内核(Renderer进程)的通信过程

1. Broswer进程接收到用户请求后，首先需要获取页面内容，如通过网络下载资源。随后将该任务通过RenderHost接口传递给Renderer进程
2. Renderer进程的RenderHost接口接收到消息，简单解释后，交给渲染线程，然后开始渲染
   1. 渲染线程接收请求，加载网页并渲染网页，这其中可能需要Broswer进程获取资源和需要GPU进程帮助渲染。
   2. 当然可能会有JS线程操作DOM(这样可能会造成回流和重绘)
   3. 最后Renderer进程将结果传递给Broswer进程

3. Broswer进程接收到结果并将结果绘制出来。

### 浏览器渲染流程

> - 浏览器输入url，浏览器主进程接管，开一个下载线程，然后进行http请求(略去DNS查询，IP寻址等等操作)，然后等待响应，获取内容，随后将内容通过RenderHost接口传给Renderer进程
> - 浏览器渲染流程开始

浏览器内核拿到内容后，渲染大概可以分为以下几个步骤：

1. 解析html和css建立dom树和style树
2. 将dom树和style树合并成render树
3. 布局render树(Layout/reflow)，负责各元素尺寸、位置的计算。
4. 绘制render树(paint)，绘制页面像素信息。
5. 浏览器会将各层的信息发送给GPU，GPU会将各层合成(composite)，显示在屏幕上。

略去了详细步骤，渲染完毕后就是load事件了，之后就是自己的JS逻辑处理。

> load事件和DOMContentLoaded事件的先后
>
> 当DOMContentLoaded事件触发时，仅当DOM加载完成，不包括样式表，图片。（建立完dom树时）
>
> 当onload事件触发时，页面上所有的DOm，样式表，脚本，图片都已经加载好了。(绘制完render树时)
>
> 所以顺序是：DOMContentLoaded —> load
